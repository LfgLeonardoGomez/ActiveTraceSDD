# Communication — Spec

> Communication preview modal for selected at-risk students, batch send with confirmation, real-time tracking via polling, lote approval/cancellation, individual message cancel/retry, and state machine display.

---

## REQ-CO-01: Communication preview shows personalized message before sending

When the user selects at-risk students and triggers the communication flow, a modal shows a preview of the message subject and body as the recipient would see it, generated by `POST /api/comunicaciones/preview`.

### Scenarios

**Scenario 1: Preview modal opens with personalized content**
GIVEN the user selected 3 at-risk students from the table
WHEN the user clicks "Comunicar seleccionados (3)"
THEN a modal/drawer opens with title "Previsualización de comunicación"
AND a summary "Se enviará a 3 destinatarios" is displayed
AND the subject line is shown: "{asunto}" as returned by the API
AND the body preview is shown with personalized fields (student name, etc.)
AND the modal shows "Enviar a {N} destinatarios" and "Cancelar" buttons
AND an "Editar" button allows modifying the template before sending

**Scenario 2: Preview generation fails**
GIVEN the user clicked "Comunicar seleccionados"
WHEN `POST /api/comunicaciones/preview` returns an error
THEN an error message "Error al generar la previsualización" is shown inside the modal
AND a "Reintentar" button retries the preview call
AND the "Enviar" button is disabled

**Scenario 3: Loading state for preview**
GIVEN the user clicked "Comunicar seleccionados"
WHILE the POST request for preview is in flight
THEN a spinner is shown inside the modal
AND the "Enviar" button is disabled
AND the text "Generando previsualización..." is displayed

**Scenario 4: Edit subject and body before sending**
GIVEN the preview modal is displayed with subject and body
WHEN the user clicks "Editar"
THEN the subject and body become editable text fields (pre-populated with preview content)
AND the modal shows "Guardar cambios" and "Cancelar edición" buttons
WHEN the user modifies the subject and clicks "Guardar cambios"
THEN the updated content is shown in the preview
AND the "Enviar" button reflects the edited content

**Scenario 5: Cancel preview and return**
GIVEN the preview modal is open
WHEN the user clicks "Cancelar"
THEN the modal closes
AND no API call for sending is made
AND the student selection in the atrasados table remains intact

## REQ-CO-02: Batch send with confirmation triggers communication lote

After preview confirmation, the user triggers `POST /api/comunicaciones/lote`. The lot enters Pendiente state and the UI transitions to tracking mode.

### Scenarios

**Scenario 1: Batch send succeeds**
GIVEN the preview modal is showing with 3 recipients
WHEN the user clicks "Enviar a 3 destinatarios"
THEN `POST /api/comunicaciones/lote` is called with `{ materia_id, alumno_ids: [...], asunto, cuerpo }`
AND on 200 with `{ lote_id }`, a success toast "Comunicación enviada a {N} destinatarios" is shown
AND the modal closes
AND the UI automatically transitions to the tracking view for that lote_id
AND the atrasados student selection is cleared

**Scenario 2: Batch send fails**
GIVEN the preview modal is showing
WHEN the user clicks "Enviar"
AND `POST /api/comunicaciones/lote` returns an error
THEN an error toast "Error al enviar la comunicación. Intentá de nuevo." is shown
AND the modal stays open
AND the "Enviar" button remains enabled for retry

**Scenario 3: Loading state during batch send**
GIVEN the user clicked "Enviar"
WHILE the POST request is in flight
THEN the "Enviar" button shows a spinner and is disabled
AND the "Cancelar" button is disabled
AND the text "Enviando comunicación..." is displayed

## REQ-CO-03: Real-time tracking polls lote state every 5 seconds

Once a communication lot is created, the ComunicacionTracking component polls `GET /api/comunicaciones/lote/{lote_id}/estado` every 5 seconds while the lote is in a non-terminal state (Pendiente/Enviando).

### Scenarios

**Scenario 1: Polling displays real-time state progression**
GIVEN a lote has been created with state `pendiente`
WHEN the tracking component renders
THEN `GET /api/comunicaciones/lote/{lote_id}/estado` is called immediately
AND `refetchInterval: 5000` is active (TanStack Query polling)
AND a summary is displayed: "{total} mensajes — {pendientes} pendientes, {enviando} enviando, {enviados} enviados, {fallidos} fallidos"
AND each message in the lote is shown in a list with its state badge
AND the polling continues while `lote.estado` is `pendiente` or `enviando`

**Scenario 2: Polling stops when lote reaches terminal state**
GIVEN the tracking component is polling a lote
WHEN the lote transitions to `enviado` (all messages terminal)
THEN the polling stops automatically
AND a success message "Comunicación completada" is shown
AND final counts per state are displayed
AND a "Volver" button returns to the atrasados tab

**Scenario 3: Polling stops when lote is cancelled**
GIVEN the tracking component is polling a lote
WHEN the lote transitions to `cancelado`
THEN polling stops
AND the state is shown as "Cancelado" with a warning icon
AND a "Volver" button is shown

**Scenario 4: Individual message states displayed with badges**
GIVEN the tracking component displays the lote items
WHEN items have different states
THEN each item shows:
- Recipient name and email
- A state badge: "Pendiente" (gray), "Enviando" (blue spinner), "Enviado" (green check), "Fallido" (red X), "Cancelado" (orange)
- If "Fallido", a "Reintentar" button is shown
- If "Pendiente" or "Enviando", a "Cancelar" button is shown (individual cancel)

**Scenario 5: User navigates away and returns**
GIVEN the tracking component is polling a lote
WHEN the user navigates to another tab and returns
THEN the lote query is still cached (TanStack Query)
AND polling resumes if the lote was not in terminal state on last check
AND the UI updates to the latest state without flickering

**Scenario 6: Error during polling**
GIVEN the tracking component is polling
WHEN a poll request fails (network error)
THEN the component retries on the next polling interval (TanStack Query handles retries)
AND a subtle warning "Actualizando..." is shown at the top
AND existing state data is preserved

## REQ-CO-04: Lote approval and cancellation (approver role)

Users with `comunicacion:aprobar` permission can approve or cancel entire lotes that are in Pendiente state.

### Scenarios

**Scenario 1: Approve pending lote**
GIVEN the authenticated user has `comunicacion:aprobar` permission
AND there is a lote in `pendiente` state
WHEN the user navigates to the lote tracking view
THEN "Aprobar lote" and "Cancelar lote" buttons are visible
WHEN the user clicks "Aprobar lote"
THEN a confirmation dialog appears: "¿Aprobar el envío a {N} destinatarios?"
WHEN the user confirms
THEN `POST /api/comunicaciones/lote/{lote_id}/aprobar` is called
AND on success, a toast "Lote aprobado" is shown
AND the lote transitions to `enviando` state
AND the approval buttons are hidden (replaced by tracking view)

**Scenario 2: Cancel pending lote (approver)**
GIVEN the authenticated user has `comunicacion:aprobar` permission
AND there is a lote in `pendiente` state
WHEN the user clicks "Cancelar lote"
THEN a confirmation dialog appears: "¿Cancelar el envío a {N} destinatarios? Esta acción no se puede deshacer."
WHEN the user confirms
THEN `POST /api/comunicaciones/lote/{lote_id}/cancelar` is called
AND on success, a toast "Lote cancelado" is shown
AND the lote state transitions to `cancelado`
AND all messages in the lote show "Cancelado" badge

**Scenario 3: Lote approval fails**
GIVEN the user confirmed lote approval
WHEN `POST /api/comunicaciones/lote/{lote_id}/aprobar` returns an error
THEN an error toast "Error al aprobar el lote. Intentá de nuevo." is shown
AND the lote remains in `pendiente` state
AND the approval/cancellation buttons remain visible

**Scenario 4: Non-approver user does not see approval actions**
GIVEN the authenticated user does NOT have `comunicacion:aprobar`
WHEN the user navigates to the tracking view
THEN no "Aprobar lote" or "Cancelar lote" buttons are rendered
AND the user only sees the read-only tracking state

## REQ-CO-05: Individual message cancel and retry

Users can cancel individual pending messages or retry failed messages directly from the tracking view.

### Scenarios

**Scenario 1: Cancel individual pending message**
GIVEN the lote tracking shows a message in `pendiente` state
WHEN the user clicks "Cancelar" on that message
THEN a confirmation prompt appears: "¿Cancelar el envío a {alumno}?"
WHEN the user confirms
THEN `POST /api/comunicaciones/{comunicacion_id}/cancelar` is called
AND on success, that message's badge changes to "Cancelado"
AND the toast "Mensaje cancelado" is shown
AND the lote summary counters update

**Scenario 2: Retry individual failed message**
GIVEN the lote tracking shows a message in `fallido` state
WHEN the user clicks "Reintentar" on that message
THEN `POST /api/comunicaciones/{comunicacion_id}/retry` is called
AND on success, that message's badge changes back to "pendiente"
AND a toast "Reintentando envío" is shown
AND the lote starts polling again if it had stopped

**Scenario 3: Retry fails**
GIVEN the user clicked "Reintentar" on a failed message
WHEN `POST /api/comunicaciones/{comunicacion_id}/retry` returns an error
THEN an error toast "Error al reintentar el envío" is shown
AND the message remains in `fallido` state

**Scenario 4: Cancel individual message fails**
GIVEN the user confirmed individual cancellation
WHEN `POST /api/comunicaciones/{comunicacion_id}/cancelar` returns an error
THEN an error toast "Error al cancelar el mensaje" is shown
AND the message state remains unchanged
AND the cancel button remains available

## REQ-CO-06: State machine display shows message lifecycle

The tracking component visually represents the Pendiente → Enviando → Enviado/Fallido/Cancelado state machine for each message and the overall lote.

### Scenarios

**Scenario 1: Lote in Pendiente state (initial)**
GIVEN a lote has just been created
WHEN the tracking view renders
THEN the header shows "Estado: Pendiente" with a gray badge
AND each message shows "Pendiente" with a gray badge and clock icon
AND a progress bar shows 0% completed

**Scenario 2: Lote transitioning (mixed states)**
GIVEN the lote is being processed
WHEN some messages are in different states
THEN the header shows "Estado: Enviando" with a blue badge and spinner icon
AND each message shows its individual state badge
AND a progress bar shows "% completado" (enviados + fallidos + cancelados) / total

**Scenario 3: Lote fully delivered**
GIVEN all messages have been sent
WHEN the tracking view renders
THEN the header shows "Estado: Enviado" with a green badge
AND the progress bar shows 100%
AND "Todos los mensajes fueron enviados correctamente" is shown

**Scenario 4: Lote with mixed terminal states**
GIVEN the lote processing is complete
WHEN some messages are `enviado`, some `fallido`, and some `cancelado`
THEN the header shows "Estado: Completado" (parcial) with a mixed-color badge
AND the summary shows: "{N} enviados, {M} fallidos, {K} cancelados"
AND a warning "Algunos mensajes no se entregaron" is shown if fallidos > 0

**Scenario 5: Lote cancelled by approver**
GIVEN the lote was cancelled before processing
WHEN the tracking view renders
THEN the header shows "Estado: Cancelado" with an orange badge
AND ALL messages show "Cancelado"
AND the text "Lote cancelado por el aprobador" is shown
